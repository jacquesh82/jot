# jot mobile (Android)

Flutter app that talks to a jot server over HTTPS/WS and reuses the Rust
crypto core (`crates/core`) through a thin FFI bridge (`crates/mobile-ffi`).

```
mobile/
├── lib/                  ← Dart source
│   ├── main.dart
│   ├── router.dart       ← go_router routes
│   ├── theme.dart
│   ├── state/            ← riverpod providers
│   ├── models/           ← Board / Note / Block / Device / Identity
│   ├── services/
│   │   ├── api_client.dart    ← dio-based HTTP client (all routes)
│   │   ├── crypto.dart        ← Dart fallback (pkg:cryptography)
│   │   ├── ffi/
│   │   │   └── bindings.dart  ← dart:ffi → libjot_mobile_ffi.so
│   │   ├── secure_storage.dart
│   │   └── ws.dart            ← WebSocket subscription
│   ├── screens/          ← Server / Onboarding / Link / Boards / Notes / Note / Devices / Profile / Settings
│   └── widgets/
├── rust/
│   └── jot_mobile_ffi.h  ← C ABI consumed by ffigen
├── android/              ← (generated by `flutter create .`)
├── pubspec.yaml
└── README.md             ← you are here
```

## First-time setup

The platform-specific scaffolding (Gradle, Manifest, MainActivity, …) is
**not** committed: Flutter regenerates it from `pubspec.yaml` and is sensitive
to local SDK versions. Bootstrap it once on your dev machine:

```bash
cd mobile

# 1. Generate android/ + ios/ scaffolding (keeps existing lib/ untouched).
flutter create . --project-name jot_mobile --org dev.jot --platforms android,ios

# 2. Pull pub dependencies.
flutter pub get
```

## Build the Rust FFI shim for Android

The mobile app loads `libjot_mobile_ffi.so` from the APK's `jniLibs`
directory. Cross-compile once per ABI you care about:

```bash
# from the repo root
rustup target add aarch64-linux-android armv7-linux-androideabi x86_64-linux-android

# Requires NDK; cargo-ndk is the easiest path.
cargo install cargo-ndk
cargo ndk -t arm64-v8a -t armeabi-v7a -t x86_64 \
  -o mobile/android/app/src/main/jniLibs \
  build -p jot-mobile-ffi --release
```

After this, `flutter run` picks up `libjot_mobile_ffi.so` automatically.

If you skip the FFI build, the app still runs — `lib/services/crypto.dart`
provides a pure-Dart fallback that matches the same algorithms (X25519 +
HKDF-SHA256 + AES-256-GCM). The FFI path is preferred for the share workflow
because it shares its randomness source with the Rust core.

## Run

```bash
# Android device or emulator
flutter run

# Specific device
flutter devices
flutter run -d <id>
```

For the emulator, the server URL defaults to `http://10.0.2.2:3000` (the
emulator's loopback to the host machine running `jot serve`).

## Surface parity (vs. SPA)

| Surface         | Status | Notes |
|-----------------|--------|-------|
| Connect to server | ✅ | `/server` screen |
| Register (open / invite) | ✅ | `/onboarding` |
| Link device (QR + manual token) | ✅ | `/link` |
| Boards list / create | ✅ | |
| Notes list / quick-add | ✅ | block editor TODO |
| Note read (E2E decrypt) | ✅ | |
| Note edit (block tree) | 🟡 | block tree shown read-only |
| Devices list / delete | ✅ | rename TODO |
| Share note / board | 🟡 | API client wired, UI panel TODO |
| Profile + logout | ✅ | |
| Export | 🟡 | API hook in place, UI TODO |
| Voice / image notes | ❌ | follows SPA roadmap |
| WebSocket live updates | 🟡 | `ws.dart` ready, screens don't subscribe yet |

The Dart API client in `lib/services/api_client.dart` already covers every
route mounted on the Axum server — so the gap is purely UI work, not protocol
work.
